- 문서 버전: v1.0
- 작성일: 2026-07-10
- 작성자: 웹 서비스 기획자 (AI)

## 1. 프로젝트 개요

### 1.1 서비스명 (가칭)
**타입파인드(TypeFind)** — "12개의 질문으로 만나는 나의 MBTI"

### 1.2 한 줄 소개
누구나 3분 안에, 12개의 짧은 질문만으로 자신의 MBTI 유형을 확인하고 결과를 예쁘게 저장·공유할 수 있는 웹 서비스.

### 1.3 배경 및 목적
MBTI는 이미 대중적으로 널리 알려진 성격 유형 프레임워크이며, SNS에서 자기소개 수단이자 대화 소재로 소비되는 콘텐츠다. 그러나 기존 MBTI 테스트 서비스들은 다음과 같은 문제를 가지고 있다.

- 질문 수가 60~90개로 지나치게 길어 완주율이 낮음
- 결과 화면이 텍스트 위주로 구성되어 공유 매력도가 낮음
- 회원가입을 요구해 진입 장벽이 있음
- 결과 저장/공유 기능이 없거나 캡처에 의존함

본 서비스는 **"짧고, 가볍고, 공유하기 좋은"** MBTI 테스트를 목표로 하며, 12문항이라는 압축된 구성과 SNS 공유 최적화된 결과 카드를 핵심 차별점으로 삼는다.

### 1.4 해결하려는 문제
- 사용자: "MBTI 테스트를 해보고 싶은데 매번 너무 길고 귀찮다"
- 사용자: "결과를 예쁘게 공유하고 싶은데 캡처하면 화면이 지저분하다"
- 사용자: "내 유형이 실제로 얼마나 흔한지/희귀한지 궁금하다"

### 1.5 핵심 가치 제안 (Value Proposition)
| 가치 | 설명 |
|---|---|
| 빠름 | 12문항, 평균 소요 시간 2~3분 |
| 간편함 | 회원가입 없이 즉시 시작 |
| 공유 친화적 | SNS 공유를 염두에 둔 결과 카드 디자인 |
| 재미 요소 | 실시간 유형 분포 통계로 "나는 얼마나 특별한가"를 확인 |

### 1.6 서비스 형태
- 무료 서비스, 회원가입 불필요 (기본값으로 설정, 추후 확장 시 선택적 로그인 고려)
- 모바일 우선 반응형 웹 (실제 트래픽의 대다수가 모바일 SNS 유입일 것으로 가정)
- 별도 앱 없이 웹으로만 제공 (PWA 확장은 추후 검토)

> **[확인 필요]** 아래 항목은 사용자 확인이 필요하나, 우선 기본값으로 진행함. 추후 방향이 다르면 알려주시면 반영하겠습니다.
> 1. 광고/수익 모델 필요 여부 (기본값: 없음, 순수 바이럴 목적)
> 2. 다국어 지원 필요 여부 (기본값: 한국어 단일 언어)
> 3. 통계 데이터를 실제 서버 DB에 누적할지, 혹은 초기 버전은 목업 데이터로 대체할지 (기본값: 실제 DB 누적, 아래 기술 스택/데이터 구조에 반영)

## 2. 타깃 사용자 분석

### 2.1 주요 페르소나

**페르소나 A: "심심풀이 재인" (핵심 타깃)**
- 연령: 16~29세, 학생 및 사회초년생
- 특성: SNS(인스타그램 스토리, 카카오톡 오픈채팅) 활동이 활발하고, 친구들과 MBTI 이야기를 자주 나눔
- 니즈: 짧은 시간에 결과를 얻고, 결과를 예쁘게 공유해서 반응을 얻고 싶어함
- 페인포인트: 기존 60문항 테스트는 지루해서 끝까지 안 함

**페르소나 B: "궁금이 지현" (2차 타깃)**
- 연령: 25~40세, 직장인
- 특성: 팀 빌딩, 소개팅, 자기이해 목적으로 MBTI를 활용
- 니즈: 결과와 함께 강점/약점/추천 직업 등 실질적 정보를 얻고 싶어함
- 페인포인트: 결과만 던져주고 설명이 부실한 서비스에 아쉬움을 느낌

### 2.2 경쟁 서비스 대비 차별점
| 구분 | 기존 서비스(예: 16personalities 등) | 본 서비스 |
|---|---|---|
| 문항 수 | 60~90문항 | 12문항 |
| 소요 시간 | 10~15분 | 2~3분 |
| 결과 공유 | 캡처 또는 링크 공유 | 전용 결과 카드 이미지 자동 생성 + 원클릭 공유 |
| 통계 | 없음 또는 제한적 | 실시간 유형 분포 통계 제공 |
| 로그인 | 일부 필요 | 불필요 |

## 4. 사용자 플로우

### 4.1 메인 시나리오: 테스트 응시 → 결과 확인 → 공유

```
[랜딩 페이지]
   │ "테스트 시작하기" 클릭
   ▼
[테스트 페이지 - Q1]
   │ 2지선다 답변 선택 (자동으로 다음 문항 전환, 별도 "다음" 버튼 없음)
   ▼
[테스트 페이지 - Q2 ~ Q11] (동일 패턴 반복, 프로그레스 바 증가)
   ▼
[테스트 페이지 - Q12]
   │ 마지막 답변 선택
   ▼
[결과 계산 중 - 로딩 인터스티셜] (1.5~2초, 기대감 조성용 연출)
   ▼
[결과 페이지]
   │ MBTI 유형 + 특징/강점/약점/추천 직업 확인
   ├─▶ "이미지로 저장" 클릭 → 결과 카드 PNG 다운로드
   ├─▶ "공유하기" 클릭 → 공유 옵션 바텀시트(카카오톡/URL복사/인스타 스토리)
   ├─▶ "우리 사이트 방문자들의 유형 분포 보기" 클릭 → 통계 섹션으로 스크롤/이동
   └─▶ "다시 테스트하기" 클릭 → 랜딩 페이지 또는 Q1로 이동
```

### 4.2 예외/분기 처리

| 상황 | 처리 방식 |
|---|---|
| 테스트 중 이탈 후 재방문 | 진행 상태 저장 안 함(기본값) — 처음부터 재시작. 단, 로컬스토리지에 "마지막 결과"는 보관하여 재방문 시 "이전 결과 다시 보기" 옵션 제공 |
| 네트워크 오류로 결과 계산 실패 | "결과를 불러오지 못했습니다. 다시 시도해주세요" 토스트 + 재시도 버튼. 응답 데이터는 로컬에 유지되어 재계산만 재시도 |
| 이미지 저장 시 브라우저 미지원(구형 브라우저) | 다운로드 실패 시 "화면을 캡처해주세요" 안내 문구로 폴백 |
| 카카오톡 SDK 로드 실패 | 공유 옵션에서 카카오톡 버튼 비활성화 처리, URL 복사는 항상 가능하도록 유지 |
| 동일 축 답변이 정확히 동점일 경우 | 데이터 구조/채점 로직(6.3) 참조 — 타이브레이커 규칙 적용 |

### 4.3 보조 시나리오: 통계 확인만 원하는 사용자
```
[랜딩 페이지] → 하단 "실시간 유형 분포 보기" 링크 클릭 → [통계 페이지] (테스트 미응시자도 열람 가능)
```

## 5. UI/UX 요구 사항

### 5.1 디자인 방향성 / 톤앤매너
- **비주얼 톤**: 밝고 친근한 파스텔톤 + 큼직한 타이포그래피. MZ세대 타깃을 고려해 과도한 정보 밀도를 피하고 "카드형 UI" 중심으로 구성
- **1화면 1액션 원칙**: 테스트 진행 중에는 한 화면에 하나의 질문만 노출, 불필요한 UI 요소 최소화
- **모바일 퍼스트**: 375px 기준으로 설계 후 태블릿/데스크톱으로 확장. 데스크톱에서도 중앙 정렬된 카드(max-width 480px)로 모바일과 동일한 경험 유지

### 5.2 화면별 상세 요구 사항

#### 5.2.1 랜딩 페이지
- 상단: 서비스 로고/타이틀 + 한 줄 소개 카피 ("12개 질문으로 알아보는 나의 진짜 성격")
- 중앙: 16유형 아이콘을 흩뿌린 배경 일러스트 또는 애니메이션 (가벼운 CSS 애니메이션, 로딩 지연 없도록 경량 처리)
- CTA 버튼: "테스트 시작하기" — 화면 내 가장 눈에 띄는 강조 색상, 하단 고정(sticky) 또는 화면 중앙 배치
- 하단: "지금까지 OOO명이 테스트했어요" 카운터(실시간 통계 데이터 연동) + "유형 분포 보기" 링크
- 소요 예상 시간 뱃지 노출: "⏱ 약 2분 소요"

#### 5.2.2 테스트 진행 페이지
- **상단 고정 영역**: 프로그레스 바
  - 형태: 전체 너비의 얇은(4~6px) 바, 현재 문항 수에 비례해 채워짐 (예: 5/12 → 41.6%)
  - 프로그레스 바 위/아래에 텍스트 표기: "5 / 12"
  - 애니메이션: 답변 선택 시 부드럽게(ease-out, 300ms) 다음 단계로 채워짐
- **중앙 영역**: 질문 카드
  - 질문 텍스트 (18~20px, 굵게)
  - 답변 선택지 2개를 세로로 배치된 큰 버튼(카드형)으로 제공 — 터치 타겟 최소 48px 높이 확보
  - 선택 시 해당 버튼에 시각적 피드백(색상 변경 + 짧은 스케일 애니메이션) 후 자동으로 다음 질문 전환 (300~400ms 딜레이)
  - "다음" 버튼 별도 없이 선택=즉시 진행 방식으로 이탈률 최소화
- **하단 영역**: "이전 질문으로" 텍스트 링크(작게, 눈에 덜 띄게 배치 — 실수 클릭 방지 목적으로 2차 기능이지만 UI 자리만 확보)
- **전환 애니메이션**: 질문 간 전환 시 슬라이드(좌→우) 트랜지션으로 진행감 부여
- **이탈 방지**: 브라우저 뒤로가기/새로고침 시 "테스트를 종료하시겠습니까?" 확인 다이얼로그(beforeunload)

#### 5.2.3 결과 계산 로딩 화면
- 심플한 스피너 또는 캐릭터 애니메이션 + "당신의 성격 유형을 분석하고 있어요..." 문구
- 최소 노출 시간 1.5초 확보(계산이 즉시 끝나도 급작스러운 전환 방지, 기대감 조성)

#### 5.2.4 결과 페이지
- **결과 카드(공유용 핵심 컴포넌트)**: 화면 상단에 배치, 별도 컨테이너로 분리해 이미지 캡처 대상이 명확하도록 설계
  - MBTI 4글자 대형 타이포 (예: "ENFP")
  - 유형 애칭 (예: "재기발랄한 활동가")
  - 유형 대표 일러스트/이모지
  - 배경: 유형별 테마 컬러 그라디언트 (16유형 각각 고유 컬러 매핑)
- **탭 또는 아코디언 구성**: "특징 | 강점 | 약점 | 추천 직업" 4개 섹션을 탭으로 전환하거나 세로 아코디언으로 순차 노출
  - 특징: 3~5개 핵심 문장
  - 강점: 체크 아이콘과 함께 리스트 3~5개
  - 약점: 주의 아이콘과 함께 리스트 3~5개 (부정적 어조 대신 "성장 포인트"로 완곡하게 표현)
  - 추천 직업: 태그/칩 형태로 5~8개 나열
- **액션 버튼 영역** (결과 카드 바로 아래, 항상 노출):
  - "이미지로 저장" (다운로드 아이콘)
  - "공유하기" (공유 아이콘) → 클릭 시 바텀시트 노출: [카카오톡 공유] [인스타 스토리 공유] [URL 복사] [기타(Web Share API)]
  - "다시 테스트하기" (보조 버튼, 톤 낮춰서 배치)
- **하단**: 실시간 유형 분포 미니 차트 삽입 — "너와 같은 유형은 전체의 N%예요" 문구 + 내 유형이 강조된 바 차트

#### 5.2.5 통계 페이지
- 16유형 전체에 대한 막대 그래프 또는 도넛 차트 (비율 %와 응답 수 함께 표기)
- 정렬 기준: 비율 높은 순 기본 정렬, "E/I, S/N, T/F, J/P 축별 비율 보기" 토글 옵션 제공
- 상단에 "누적 참여자 수" 대형 숫자 노출

### 5.3 인터랙션 / 마이크로 인터랙션
- 답변 버튼 탭 시: 짧은 스케일(0.98배) + 색상 전환 애니메이션 (haptic feedback은 모바일 브라우저 제약상 제외)
- 프로그레스 바: 채워지는 애니메이션이 답변 선택과 동기화되어 "진행되고 있다"는 감각 제공
- 결과 페이지 진입 시: 결과 카드가 아래에서 위로 슬라이드 업 + 페이드 인
- 공유 완료 시: 짧은 토스트 메시지 "공유되었습니다!"

### 5.4 반응형 / 접근성
- 반응형 브레이크포인트: 모바일(~480px) / 태블릿(481~1024px) / 데스크톱(1025px~)
- 데스크톱에서는 콘텐츠 영역을 카드형으로 중앙 정렬(max-width 480~560px)하여 모바일과 톤을 통일, 좌우 여백에는 은은한 배경 패턴만 배치
- 색상 대비 WCAG AA 기준 충족 (텍스트 대비 4.5:1 이상)
- 답변 버튼에 `aria-label` 부여, 스크린 리더 대응 (키보드 Tab/Enter로 답변 선택 가능하도록 처리)
- 프로그레스 바에 `role="progressbar"` 및 `aria-valuenow` 속성 부여

## 7. 데이터 구조

### 7.1 핵심 엔티티

#### `questions` (질문)
| 필드 | 타입 | 설명 |
|---|---|---|
| id | INT (PK) | 질문 고유 ID (1~12) |
| axis | ENUM('EI','SN','TF','JP') | 판별 축 |
| order | INT | 노출 순서 |
| question_text | TEXT | 질문 본문 |
| option_a_text | TEXT | 선택지 A 텍스트 |
| option_a_polarity | ENUM('E','I','S','N','T','F','J','P') | 선택지 A 선택 시 부여되는 극성 |
| option_b_text | TEXT | 선택지 B 텍스트 |
| option_b_polarity | ENUM('E','I','S','N','T','F','J','P') | 선택지 B 선택 시 부여되는 극성 |

- 총 12행: EI 축 3문항, SN 축 3문항, TF 축 3문항, JP 축 3문항으로 시드 데이터 구성

#### `mbti_types` (16유형 마스터 데이터)
| 필드 | 타입 | 설명 |
|---|---|---|
| code | VARCHAR(4) (PK) | 예: 'ENFP' |
| nickname | VARCHAR | 유형 애칭 (예: '재기발랄한 활동가') |
| description | TEXT | 특징 설명 |
| strengths | JSON/TEXT[] | 강점 리스트 |
| weaknesses | JSON/TEXT[] | 약점(성장 포인트) 리스트 |
| recommended_jobs | JSON/TEXT[] | 추천 직업 리스트 |
| theme_color | VARCHAR | 결과 카드 테마 컬러 코드 |
| illustration_url | VARCHAR | 대표 일러스트 이미지 경로 |

#### `test_results` (응시 결과 — 통계 집계용)
| 필드 | 타입 | 설명 |
|---|---|---|
| id | UUID (PK) | 응답 고유 ID |
| result_type | VARCHAR(4) (FK → mbti_types.code) | 최종 판정된 MBTI 유형 |
| ei_score | SMALLINT | E/I 축 점수 (예: E 방향 +값, I 방향 -값 혹은 0~3 카운트) |
| sn_score | SMALLINT | S/N 축 점수 |
| tf_score | SMALLINT | T/F 축 점수 |
| jp_score | SMALLINT | J/P 축 점수 |
| created_at | TIMESTAMP | 응시 시각 |
| session_id | VARCHAR (nullable) | 익명 세션 식별자(중복 집계 방지용, 쿠키 기반) |

> 개인 식별 정보는 저장하지 않으며, `test_results`는 순수 통계 집계 목적으로만 사용한다.

### 7.2 엔티티 관계 (ERD 요약)
```
questions (12건, 고정 시드 데이터)
   └─ axis(EI/SN/TF/JP) 기준으로 test_results 채점 로직에 참조됨 (직접 FK 관계 없음)

mbti_types (16건, 고정 시드 데이터)
   └─ 1 : N → test_results.result_type
        (하나의 유형은 여러 응시 결과를 가질 수 있음)
```

### 7.3 채점 로직 (Scoring Logic)
1. 사용자가 12문항에 응답하면, 각 응답의 `polarity`(E/I/S/N/T/F/J/P) 값을 수집한다.
2. 축(EI, SN, TF, JP)별로 3문항이 배정되어 있으므로, 각 축마다 두 극성 중 더 많이 선택된 쪽이 채택된다.
   - 예: EI 축 3문항 중 E 2회, I 1회 선택 → 최종 E 판정
3. **동점 처리(타이브레이커)**: 각 축은 3문항(홀수)으로 구성되어 원칙적으로 동점이 발생하지 않도록 설계한다.
   - 단, 향후 문항 수 조정 등으로 짝수 배정이 되는 예외 상황을 대비해 축별 "기본 극성"(예: EI 동점 시 I 우선)을 사전 정의해 폴백 처리한다.
4. 4개 축의 최종 극성을 조합해 4글자 코드(예: `ENFP`)를 생성하고, `mbti_types` 테이블에서 해당 코드의 상세 데이터를 조회한다.
5. 계산이 완료되면 `test_results`에 익명 레코드 1건을 INSERT하여 통계에 반영한다.

## 8. 개발 우선순위 (로드맵)

### 8.1 1단계: MVP (출시 목표)
- 랜딩 페이지
- 12문항 테스트 + 프로그레스 바
- 결과 계산 로직 + 결과 페이지(특징/강점/약점/추천 직업)
- 결과 이미지 저장 기능
- URL 복사 공유 + Web Share API 공유
- 유형 분포 통계 페이지(기본 막대 차트)
- 근거: 핵심 가치 제안("빠르고 간편하고 공유 가능한 테스트")을 검증하기 위한 최소 기능 집합. 카카오 공유 등 부가 연동 없이도 바이럴 효과를 1차 검증 가능

### 8.2 2단계: 1차 고도화
- 카카오톡 공유 SDK 연동
- 이전 질문으로 돌아가기(답변 수정) 기능
- 결과 공유 링크 진입 시 OG 이미지 미리보기(SNS 링크 공유 최적화)
- 통계 페이지 축별(EI/SN/TF/JP) 비율 토글 뷰
- 근거: 1단계에서 검증된 트래픽을 바탕으로 SNS 공유 전환율을 극대화하는 기능에 투자

### 8.3 3단계: 2차 고도화
- 인스타그램 스토리 딥링크 공유
- 관리자 통계 대시보드(응답 추이, 유입 경로 분석)
- 다국어 지원(영어 등)
- 궁합 테스트, 유형별 콘텐츠 아티클 등 부가 콘텐츠 확장
- 근거: 서비스 안정화 이후 리텐션 및 신규 유입 채널 다각화를 위한 확장 기능
